# 2026-03-22 — Daily Work Log

## Project: ProposaIQ — Proposal Generator for Freelancers

## Overview

Full engineering session covering six areas: input quality signals, output
validation hardening, UI consistency, text rendering correctness, operational
safeguards, and Upstash Redis setup.

---

## 1. Pre-processing and Input Signal Layer

**Files:** `lib/server/generation/preprocessor.ts`, `lib/domain/proposal/signals.ts`

- Created a unified `analyzeRequirement()` function in `signals.ts` as the single
  source of truth for requirement signal detection.
- Detects budget mentions, deadline mentions, scope vagueness, and
  budget/timeline conflicts using exported regex pattern arrays.
- `preprocessor.ts` reduced to a two-line re-export so the server adapter
  cannot drift from the client layer.
- `buildUserMessage()` in `prompt.ts` extended to accept `RequirementSignals`
  and inject a `[CONTEXT SIGNALS]` block into the model prompt, giving the
  model explicit context on missing budget, missing deadline, vagueness level,
  and detected conflicts.

---

## 2. Client-Side Input Warnings

**Files:** `lib/domain/proposal/warnings.ts`, `app/page.tsx`, `app/globals.css`

- `computeWarnings()` runs debounced (600 ms) in the browser as the user types.
- Imports patterns directly from `signals.ts` -- browser warnings and server
  prompt hints share identical detection logic and cannot drift.
- Three severity levels: `info` (blue-grey), `caution` (amber), `conflict` (red).
- Warnings render below the textarea, hidden during generation.
- CSS added: `.warnings-list`, `.warning-badge`, `.warning-badge--{severity}`,
  `.warning-icon`, `.warning-label`, `.warning-detail` with fade-in animation.

---

## 3. Zod Schema Validation

**Files:** `lib/domain/proposal/proposalSchema.ts`, `lib/server/generation/validator.ts`

- Replaced handwritten structural validator with a single Zod schema.
- Schema validates nested object shapes and array item types, not just
  top-level presence.
- Semantic rules enforced inside the schema via `.refine()` and `.transform()`:
  - `scope.core` must have at least 1 item
  - `pricing.modules` must have at least 2 entries
  - `pricing.rationale` must be at least 30 characters
  - `timeline.phases` must have at least 1 entry
  - `feasibility` orange/red requires a non-null `feasibilityNote`
  - `pricing.totalMax >= pricing.totalMin`
  - `timeline.totalDays` derived from phase sums (model value ignored)
  - Blank strings filtered from all string arrays
  - All strings trimmed during parse
  - Numbers clamped to guardrail bounds
- `validateProposal()` now returns a discriminated union:
  `{ success: true, data: Proposal }` or `{ success: false, reason: string }`.
- `normalizeProposal()` no longer called separately -- Zod transforms handle it.

---

## 4. Repair Retry

**Files:** `lib/server/generation/index.ts`, `lib/server/generation/prompt.ts`

- Added `buildRepairMessage(rawOutput, failureReason)` to `prompt.ts`.
  Terse: shows the model exactly what broke and what to fix.
- `generateProposal()` restructured around a private `attempt()` helper that
  returns a typed discriminated union (never throws on parse/validation failure).
- Flow: first attempt → on failure, one repair attempt with the raw output and
  failure reason → on second failure, throw `GenerationError("REPAIR_FAILED")`.
- `GenerationResult` extended with `meta: GenerationMeta` carrying latency,
  token counts, `repairUsed`, and `firstFailReason`.

---

## 5. Route and Error Hardening

**Files:** `app/api/generate/route.ts`, `lib/domain/proposal/schema.ts`,
`lib/domain/proposal/constants.ts`

- `INVALID_BODY` and `RATE_LIMITED` added to `ErrorCode` union.
- `STATUS_MAP: Record<ErrorCode, number>` replaces the dead ternary that
  mapped everything to 502. TypeScript enforces exhaustiveness.
- Malformed JSON body now correctly returns code `INVALID_BODY` (was
  incorrectly returning `INPUT_TOO_SHORT`).
- Status codes: 400 bad/short input, 413 too long, 429 rate limited,
  500 config/unexpected, 502 all upstream failures.

---

## 6. UI and Render Consistency

**Files:** `components/proposal/ProposalView.tsx`, all 11 section components,
`lib/server/rendering/text.ts`

- `renderedText` removed from `ProposalViewProps` -- it was received but never
  read inside the component.
- `VisibilityMap` interface introduced: one `computeVisibility(proposal)`
  function covers all 7 conditional sections. Nav filter and JSX conditional
  renders both read from the same `vis` object -- they cannot drift.
- `NAV_ITEMS` extended with `visKey` field linking each nav entry to its
  `VisibilityMap` key or `"always"`.
- `useMemo` added for `vis` and `visibleNav`; `useEffect` deps corrected to
  `[visibleNav]`.
- `BoundariesSection`, `DeliverablesSection`, `NextStepsSection`,
  `TechStackSection` all given `if (!items.length) return null` guards.
- `text.ts` fixed: `feasibilityNote` no longer appears twice (was rendered
  inline in Overview AND in its own section). Section numbers changed from
  hardcoded strings to a running counter so conditional sections never create
  numbering gaps or collisions.

## 7. Horizontal Scroll Fix

**File:** `app/globals.css`

- `overflow-x: hidden` on both `html` and `body`.
- `grid-template-columns` changed from `var(--input-w) 1fr` to
  `var(--input-w) minmax(0, 1fr)` -- `1fr` does not prevent grid children
  from overflowing; `minmax(0, 1fr)` does.
- `.section-nav` negative margin replaced with `width: calc(100% + 64px);
  margin-left: -32px` -- the right negative margin was adding invisible
  horizontal space and triggering the scrollbar.
- Mobile breakpoint nav margin updated to match.

## 9. Operational Safeguards

**Files:** `lib/server/ratelimit.ts`, `lib/server/logger.ts`,
`lib/server/generation/model.ts`, `package.json`

- **Rate limiting:** `checkRateLimit(ip)` with two backends:
  - Production: Upstash Redis sliding window, 5 req / 15 min per IP.
    Lazy dynamic `import()` so missing packages don't crash dev.
  - Development: in-memory fixed window with one-time console warning.
- **Structured logging:** `createRequestLog()` + `emitLog()` writes one JSON
  line per request to stdout. Fields: `requestId`, `timestamp`, `ip`, `model`,
  `latencyMs`, `inputTokens`, `outputTokens`, `repairUsed`, `failureClass`,
  `status`, plus optional `parseFailReason`, `validationFailReason`,
  `repairFailReason`. Every exit path in `route.ts` emits a log line.
- `latencyMs` added to `ModelResponse`, measured across the full Groq SDK call.
- `@upstash/ratelimit` and `@upstash/redis` added to `package.json` dependencies.
